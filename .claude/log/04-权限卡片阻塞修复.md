# 04 - 权限卡片阻塞修复

## 问题

Claude Code 需要工具权限时，会通过 channel 发 `permission_request` 通知到飞书。用户在飞书点按钮授权后，Claude Code 才继续。但实际使用中发现：**权限请求发出后整个 channel 卡死，后续消息全部发不过来。**

## 根因分析

两个 bug 叠加：

### Bug 1：permission 卡片发给了 chat_id

`access.allowFrom` 里同时有 `ou_xxx`（sender open_id）和 `oc_xxx`（chat_id）。权限卡片发送遍历 `allowFrom`，用 `receive_id_type: 'open_id'` 发送。`oc_xxx` 不是 open_id，飞书 API 报错。

如果 `ou_xxx` 那条也因为网络等原因失败，卡片完全发不出去 → 用户看不到授权按钮 → Claude Code 一直等 → 消息阻塞。

### Bug 2：`assertAllowedChat` 依赖已被删除的文件

`assertAllowedChat` 用 `approved/` 目录做 sender_id → chat_id 映射。但 `checkApprovals()` 在发送 "Paired!" 确认后就 **删除了 approved 文件**（`rmSync`）。

之前之所以能工作，是因为 `allowFrom` 里有 `oc_xxx` 做兜底。这是个隐藏的依赖 — 看起来是冗余数据，实际上是唯一的回复路径。

## 修复

### 1. permission 卡片过滤非 open_id

```typescript
for (const userId of access.allowFrom) {
  if (!userId.startsWith('ou_')) continue  // ← 新增
  // ... send card
}
```

### 2. 运行时 knownChats 映射替代 approved 文件

```typescript
// 全局 Map
const knownChats = new Map<string, string>()  // chat_id → sender_id

// inbound handler 中，gate 通过后记录：
knownChats.set(chatId, senderId)

// assertAllowedChat 中查询：
const mappedSender = knownChats.get(chat_id)
if (mappedSender && access.allowFrom.includes(mappedSender)) return
```

### 3. 清理 access.json

从 `allowFrom` 移除 `oc_xxx`，只保留 `ou_xxx`。不再需要 chat_id 做兜底。

## 清理 checklist

- [x] `access.json` 移除 `oc_xxx`
- [x] permission 发送跳过非 `ou_` 前缀
- [x] `assertAllowedChat` 用 `knownChats` 替代 `approved/` 文件查找
- [x] inbound handler 记录 chat→sender 映射

## 教训

1. **临时文件不能当持久存储用** — `approved/` 设计为一次性通知机制（写入 → 确认 → 删除），不该被 `assertAllowedChat` 当作查询数据源
2. **allowFrom 应该只存 open_id** — 混入 chat_id 是 workaround，会导致下游（permission 发送）出错
3. **运行时 Map 比文件系统更可靠** — 对于 "消息 A 和消息 B 属于同一个会话" 这类会话级别的映射关系
